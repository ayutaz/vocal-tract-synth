## 2. 実装する内容の詳細

### 2.1 LF声門モデルの実装（Liljencrants-Fant モデル）

Liljencrants & Fant (1985) に基づく4パラメータ声門流モデルを実装する。LFモデルは声門波形を開放相と閉鎖相の2区間で記述する。

**LF波形の数式定義:**

- 開放相 (0 <= t <= Te):
  ```
  U(t) = E0 * exp(alpha * t) * sin(omega_g * t)
  ```
  - omega_g = pi / Tp（声門開放の立ち上がり周期）
  - alpha: 振幅包絡を制御する指数パラメータ

- 閉鎖相 (Te <= t <= T0):
  ```
  U(t) = -(Ee / (epsilon * Ta)) * (exp(-epsilon * (t - Te)) - exp(-epsilon * (T0 - Te)))
  ```
  - Ee: 最大励振振幅（Te時点での流量微分の絶対値）
  - Ta: リターンフェーズの時定数（声門閉鎖の急峻さ）
  - epsilon: 1/Ta から暗黙方程式で求まる減衰係数

**暗黙方程式の数値解法:**

alpha の決定には以下の暗黙方程式を解く必要がある:

```
(1/alpha) - (1/(alpha + (omega_g^2 / alpha))) = Te - Tp
```

Newton-Raphson法による反復求解を実装する:
- 初期推定値: alpha_0 = 1 / Tp
- 収束判定: |f(alpha)| < 1e-10（10回以内に収束）
- 発散防止: alpha の上下限クランプ

epsilon の決定:
```
1 - exp(-epsilon * (T0 - Te)) = epsilon * Ta
```
同様にNewton-Raphson法で解く。

**実装場所**: `src/models/glottal-source.ts` に `LFGlottalSource` クラスを追加

**波形生成の実装方針:**
- 位相アキュムレータ方式は既存KLGLOTT88と同様（`phase += F0 / sampleRate`）
- 1周期ごとにRdパラメータからLFパラメータを再計算
- process()内ではルックアップテーブル方式も検討（計算コスト軽減）

### 2.2 Rdパラメータによる声質制御

Fant (1995) の簡易パラメータ化に基づき、単一のRdパラメータから全LFパラメータを導出する。

**Rdの意味と範囲:**

| Rd 値 | 声質 | 特徴 |
|-------|------|------|
| 0.3 - 0.7 | 押し声（Pressed） | 声門閉鎖が急峻、倍音豊富 |
| 0.7 - 1.2 | チェストボイス（Modal） | 通常の発声、バランスの取れた倍音構造 |
| 1.2 - 2.0 | 柔らかい声（Lax） | 緩やかな閉鎖、高域倍音が少ない |
| 2.0 - 2.7 | ファルセット（Breathy） | 声門閉鎖が不完全、気息成分多い |

**Rd → LFパラメータ変換式:**

```typescript
// Rd から正規化パラメータへの変換（Fant 1995）
const Rap = (-1 + 4.8 * Rd) / 100;                    // Tp/Te 比
const Rkp = (22.4 + 11.8 * Rd) / 100;                 // (Te-Tp)/Tp 比
const Rgp = 1 / (4 * ((0.11 * Rd / (0.5 + 1.2 * Rkp)) - Rap)); // 不使用の場合もあり

// 正規化パラメータ → 絶対パラメータ（T0 = 1/F0 に基づく）
const Tp = Rap * Te;                                    // 声門開放ピーク時刻
const Te = T0 * (1 / (1 + Rkp));                       // 最大励振時刻（近似）
const Ta = Rap * T0;                                    // リターンフェーズ時定数
const Ee = -1.0;                                        // 正規化（振幅はF0と連動で調整）
```

注: 上記は簡略化した変換。実装時にはFant (1995) およびDegottex et al. (2010) の厳密な変換式を参照し、数値安定性を検証すること。

**実装場所**: `src/models/glottal-source.ts` 内に `rdToLFParams()` 関数を追加

### 2.4 気息ノイズ（Aspiration）の追加


## 5. 実装に関する懸念事項とレビュー項目

### 5.1 LFモデルの暗黙方程式の数値解法

**懸念**: alpha と epsilon の決定に Newton-Raphson 法が必要であり、AudioWorklet の process() 内で反復計算を行うとリアルタイム性を損なう可能性がある。

**対策:**
- Newton-Raphson は1周期に1回のみ実行（毎サンプルではない）。F0=400Hz でも 44100/400=110 サンプルに1回。
- 初期推定値を前回の収束値から開始（warm start）することで反復回数を削減
- 最悪ケースでも10回以内に収束するよう上下限クランプを設置
- 代替案: Rd の離散グリッドで事前計算したテーブルを補間（完全に反復を排除）

**レビュー項目:**
- [ ] Newton-Raphson の収束条件と最大反復回数の妥当性
- [ ] 発散時のフォールバック処理（前回の値を使用）
- [ ] process() 内での実行時間計測結果

### 5.2 Rdパラメータ範囲外での安定性

**懸念**: Rd が極端な値（0.3未満や2.7超）に設定された場合、LFパラメータ変換式が物理的に意味のない値を返す可能性がある。

**対策:**
- UI側でスライダーの範囲を 0.3-2.7 にハードリミット
- 計算側でもクランプ: `Rd = Math.max(0.3, Math.min(2.7, Rd))`
- Tp, Te, Ta の各パラメータに独立した妥当性検証（0 < Tp < Te < T0, Ta > 0）
- 不正値検出時はデフォルト値（Rd=1.0相当）にフォールバック

**レビュー項目:**
- [ ] Rdの全範囲（0.3-2.7、0.01刻み）でのパラメータ変換結果の検証
- [ ] F0の全範囲（80-400Hz）との組み合わせでの安定性
- [ ] フォールバック処理のテストカバレッジ

### 5.3 KLGLOTT88との切替時の不連続性

**懸念**: モデル切替時に波形の不連続が発生し、クリック音やポップノイズが聞こえる可能性がある。

**対策:**
- 声門周期の開始点（phase ≈ 0）でのみモデルを切り替える
- 切替時に5-10msのクロスフェードを適用（2モデルを一時的に並行実行）
- 切替指示はフラグとして保持し、次のゼロクロスで実行

**レビュー項目:**
- [ ] 切替時のオーディオ波形のオシロスコープ検査（不連続の有無）
- [ ] 高速な切替連打での安定性
- [ ] 切替中のパラメータ状態の一貫性

### 5.4 全体のCPU負荷

**懸念**: LFモデルはKLGLOTT88より計算コストが高い（指数関数 exp() と三角関数 sin() を毎サンプル計算）。44区間KLフィルタ + LF音源 + 気息ノイズフィルタの合計負荷が process() の予算（2.9ms）を超える可能性がある。

**対策:**
- LF波形のルックアップテーブル化: 1周期分を1024点で事前計算、線形補間で読み出し
  - テーブル再計算: F0またはRdが変化した周期のみ（変化がなければテーブル再利用）
  - テーブルサイズ: Float64Array(1024) = 8KB（十分小さい）
- Math.exp() / Math.sin() の高速近似関数の検討（精度とのトレードオフ）
- 気息ノイズのバンドパスフィルタを2次IIR（biquad）に限定（計算コスト最小）

**レビュー項目:**
- [ ] process() の実行時間計測（LFモデル有効時 vs KLGLOTT88時の比較）
- [ ] ルックアップテーブル方式の音質劣化の有無（テーブルサイズの妥当性）
- [ ] 負荷ピーク時（F0最高 + Rd変更 + 断面積変更が同時発生）の挙動

### 5.5 その他のレビュー項目

- [ ] LF波形の物理的妥当性（文献値との比較、声門流量の時間積分がゼロに近いこと）
- [ ] 気息ノイズのバンドパスフィルタ係数の正当性
- [ ] UI追加によるメインスレッドの負荷増分
- [ ] postMessage の送信頻度（Rdスライダーのドラッグ中）
- [ ] 全フェーズ統合後のメモリプロファイル

## 6. 一から作り直すとしたら

### 6.1 Phase 1-5 全体を統合した設計観点

Phase 1-5 を最初から見通して設計するなら、以下の点を変更する。

#### 最初からLFモデルベースで始めるべきだったか

**結論: 段階的にKLGLOTT88 → LFが正解だった。**

理由:
- KLGLOTT88 は多項式ベースで暗黙方程式が不要。Phase 1-2 で「音が出る」状態を早期に達成できた
- LFモデルの Newton-Raphson 解法のデバッグは、声道フィルタ（KL + 壁面損失 + 放射フィルタ）が正しく動作している状態で行うほうが、音源の問題か声道の問題かの切り分けが容易。Phase 2 で壁面損失が導入済みであることが前提条件
- KLGLOTT88 がベースラインとして存在することで、LFモデルの音質をA/B比較できる
- 段階的実装により、各フェーズで動作する成果物を確認しながら進められた

この設計判断は、Phase 2 の rebuild で述べた「壁面損失を Phase 1 から入れるべきだった」という提案と整合する。壁面損失は Phase 1 から入れ、声門音源は段階的に高品質化する、というのが全体を通じた最適戦略であった

ただし改善点として:
- Phase 1 の時点で `GlottalSourceInterface` を定義しておけば、Phase 5 での切替実装がよりクリーンだった。Phase 1 の rebuild で「過度な抽象化は不要」としつつ「関数シグネチャの統一」を推奨したが、この判断は正しかった。Phase 2 で `GlottalSource` インターフェースを抽出するタイミングが最適であり、Phase 1 時点でのインターフェース定義は YAGNI（必要になるまで作らない）の範囲で十分だった
- Rdパラメータの概念をPhase 2のKLGLOTT88実装時にOQ（Open Quotient）として先取りしておけば、UIの連続性が向上した

#### AudioWorklet プロセッサの全体アーキテクチャ

最初から設計するなら:
- **コマンドパターンの導入**: postMessage で送るコマンドを型安全な判別共用体型（discriminated union）で管理。Phase が増えるごとに `if/else` が増殖するのを防ぐ。ただし AudioWorklet 内のコマンドディスパッチは、GC 回避のため `switch` 文 + プリミティブ型比較に限定し、Map/オブジェクト生成を伴うパターンは避ける
- **モジュール分離**: worklet-processor.ts 内の責務を明確に分離
  ```
  worklet-processor.ts
    ├── GlottalSource (インターフェース + 実装2種)
    ├── VocalTractFilter (KL アルゴリズム)
    ├── RadiationFilter
    └── OutputMixer (有声/無声/気息の混合)
  ```
- **状態管理の一元化**: パラメータの二重管理（UI側とWorklet側）を防ぐため、パラメータストアの設計を最初に行う
- **ダブルバッファリング**: 断面積配列の更新をダブルバッファ方式にし、process() のフレーム途中で断面積が切り替わる不整合を防ぐ。なお postMessage は構造化クローンでデータをコピーするため、メインスレッドとの競合は発生しない。ダブルバッファの目的はコピー回避ではなく、`port.onmessage` による受信タイミングと `process()` の実行タイミングの不整合を防ぐ点にある（受信は process() の外で発生し、process() 実行中に断面積が変わらないことを保証する）

#### UIコンポーネントの設計パターン

最初から設計するなら:
- **イベントバスの導入**: UI要素間の通信を publish/subscribe パターンで疎結合化。Phase 4 の rebuild で提案した `SynthEvent` / `EventScheduler` 型のイベントシステム（手動/自動/MIDI の統一インターフェース）と同一の設計思想であり、Phase 1 の時点でこの基盤を敷いておけば Phase 4 の統合コーディネータが大幅に簡素化できた
- **UIステートマシン**: アプリの状態（停止中 / 再生中 / Auto中 / パラメータ編集中）を明示的なステートマシンで管理
- **CSSカスタムプロパティの体系化**: 配色・スペーシング・フォントサイズをCSS変数で一元管理し、Phase ごとのUI追加で統一感を維持
- **Canvas描画の抽象化層**: 声道エディタとスペクトル表示で共通の描画ユーティリティを先に設計

#### テスト戦略の全体最適化

最初から設計するなら:
- **Phase 1でテストインフラを構築**: Vitest の導入と AudioWorklet のモック環境の整備。AudioWorklet の process() 内のロジック（KLアルゴリズム、音源生成）は純粋な数値計算モジュールとして分離してあるため（Phase 1 rebuild 6.2 参照）、Node.js 上でのユニットテストが可能。Canvas のスナップショットテストは費用対効果が低いため優先度は下げてよい
- **各PhaseにCI/CDゲートを設置**: Phase N の完了条件をCIで自動検証してから Phase N+1 に進む。具体的には、KL均一管の共鳴周波数テスト（Phase 1）、KLGLOTT88波形連続性テスト（Phase 2）、フォルマント計算精度テスト（Phase 3）等をCIに組み込む
- **音声品質の自動テスト**: 合成音声のスペクトル特性を自動検証するテストフレームワーク。ただし実現にはオフラインレンダリング（OfflineAudioContext）を使ったテスト用パイプラインが必要で、AudioWorklet の OfflineAudioContext 対応はブラウザ依存の制約がある。現実的には、数値計算モジュール単体でインパルス応答を生成し、FFT でフォルマントピークを検出する方式が堅実
- **パフォーマンスリグレッションテスト**: process() の実行時間をCIで計測し、閾値超過でアラート。CI環境（GitHub Actions）のCPU性能はばらつきがあるため、絶対値ではなく相対比較（KLGLOTT88比でLFモデルが何倍か）を指標とする

#### プロジェクト全体の設計思想の総括

本プロジェクトは「段階的に動くものを作り、各段階で品質を確認しながら進む」アプローチを採用した。これは source-filter model という物理モデルベースの音声合成において正しい判断であった。理由:

1. **物理モデルの検証可能性**: 各 Phase で「音が出る」「母音が聞こえる」という聴覚的な検証が可能
2. **問題の局所化**: Phase N で問題が発生した場合、Phase N-1 までは正常動作が保証されているため、原因の特定が容易
3. **技術リスクの段階的解消**: 最もリスクの高い Kelly-Lochbaum アルゴリズム（Phase 1）とLFモデル（Phase 5）を最初と最後に配置し、中間 Phase で安定基盤を構築
4. **ユーザー価値の早期提供**: Phase 2 の時点で「母音を鳴らせる楽器」として最小限の価値がある

一方、全体設計を最初に行う場合は、インターフェースの安定性と変更コストのバランスが課題となる。Vanilla TypeScript（フレームワークなし）の選択は以下のトレードオフがあった:

- **利点**: 依存パッケージゼロによるメンテナンスコストの極小化、AudioWorklet との親和性（フレームワークのランタイムが不要）、バンドルサイズの最小化、Web Audio API を直接操作する透明性
- **欠点**: UIコンポーネントの再利用性が低い（Phase 4-5 でスライダー/ボタンのパターンが重複）、DOMの手動管理が煩雑化する、状態管理を自前で設計する必要がある
- **総合判断**: 本プロジェクトのUI規模（スライダー数個 + Canvas 2枚 + ボタン群）ではフレームワーク導入のオーバーヘッドが利点を上回るため、Vanilla TypeScript が妥当。UIが大幅に複雑化する拡張（多数のプリセット管理画面、設定パネル等）を行う場合に再検討する

### 6.2 補足: レビューによる追加事項

#### 保守・運用の観点

Phase 5 はプロジェクトの最終フェーズであり、完了後の保守性を考慮した設計が重要:

- **依存パッケージの更新**: 本プロジェクトのランタイム依存はゼロ（Vanilla TypeScript + Web Audio API）。開発依存は Vite + TypeScript + Vitest のみであり、更新頻度は低い。Vite のメジャーバージョンアップ時はビルド設定の互換性を確認する程度で済む
- **ブラウザ API の変更リスク**: Web Audio API（AudioWorklet 含む）は W3C 勧告候補であり、破壊的変更のリスクは低い。ただし AudioWorklet の `addModule()` のモジュール解決仕様はブラウザ間で微差があるため、Vite のビルド方式（URL コンストラクタパターン）が将来も正しく動作するか、主要ブラウザのリリースノートを定期確認する
- **長期的な音質改善**: LFモデルのパラメータ変換式（Rd → Tp/Te/Ta）は研究が進行中の領域。Degottex (2010) 以降の改良版変換式が提案された場合、`rdToLFParams()` 関数の差し替えだけで対応できる設計にしておく

#### アクセシビリティの考慮

本アプリは声道の視覚的・聴覚的操作が主目的であり、スクリーンリーダー完全対応は現実的ではないが、以下は対応すべき:

- **キーボード操作**: Canvas 上の制御点をキーボード（Tab でフォーカス移動、矢印キーで断面積調整）で操作可能にする。これは視覚障害対応だけでなく、マウス操作より精密な調整を求めるユーザーにも有益
- **ボタン/スライダーのラベル**: 全ての `<button>` と `<input type="range">` に適切な `aria-label` を付与する。Phase 1-4 で追加した UI 要素を Phase 5 のリファインで一括整備する
- **色のコントラスト比**: CSSカスタムプロパティによるテーマ管理（6.1 の UI コンポーネント設計パターンで提案）において、WCAG 2.1 AA レベルのコントラスト比（4.5:1）を満たす配色を選択する

#### GitHub Pages デプロイの CI/CD パイプライン

セクション 7.2 で公開・デプロイの残タスクを列挙しているが、具体的な GitHub Actions ワークフローの構成を記載する:

```yaml
# .github/workflows/deploy.yml の構成概要
# トリガー: main ブランチへの push
# ステップ:
#   1. actions/checkout
#   2. actions/setup-node (Node.js LTS)
#   3. npm ci
#   4. npm run build (vite build, base設定でリポジトリ名を指定)
#   5. npm run test (Vitest によるユニットテスト)
#   6. actions/upload-pages-artifact (dist/ ディレクトリ)
#   7. actions/deploy-pages (GitHub Pages へデプロイ)
```

- テストが失敗した場合はデプロイを中断する（CI/CD ゲート）
- `vite.config.ts` の `base` は環境変数 `GITHUB_REPOSITORY` から動的に設定可能にしておくと、フォーク先でも設定変更なしでデプロイできる

### 7.1 将来拡張への準備

#### MIDI / OSC 対応への道筋
- F0 を AudioParam (k-rate) で管理しているため、MIDI CC → AudioParam の接続は容易
- Rd パラメータも AudioParam 化すれば、外部コントローラーからの制御が可能
- Web MIDI API の接続層を追加する場合、`src/midi/` ディレクトリを新設
- OSC は WebSocket 経由の受信が必要（別途サーバーが必要になるため、GitHub Pages デプロイとの整合性に注意）

#### 歌唱技法の高度化
- ビブラートの Rd 連動（ビブラート中に Rd も微小変動させると自然さが向上）
- フレーズのダイナミクスと Rd の連動（強い発声 → 低Rd、弱い発声 → 高Rd）
- これらのパラメータリンクを Auto Singer モジュールで制御可能にする拡張点を残す
